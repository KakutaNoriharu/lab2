# 2026-05-12

## 概要
シミュレーション出力の通信記録（r.csv）を読み込み、情報の伝播経路を可視化した。
偽装情報がどのノードを経由して広がるかを図で確認できるようにした。

## データ
- r.csv: 2,234 件の通信レコード
- 列: No, Time, Source, Target, Content (0=TRUE情報, 1=偽装情報)
- Content=0: 2,208 件 / Content=1: 26 件

## 分析内容

### ノード別通信集計
各ノードについて以下を集計した。
- Content=0 / Content=1 の送信回数（content_0, content_1）
- from（最初に情報を受け取った送信元）/ to（転送先ノード群）の関係
- 伝播ツリーを深さ優先探索で構築し、各ノードのY座標（伝播順）を決定

### 可視化1: 伝播経路図
- X 軸: 受信時刻、Y 軸: 伝播インデックス（初登場順）
- シアン点/線: Content=0 のみ送信したノード（正規）
- 赤点/線: Content=1 のみ送信したノード（攻撃者候補）
- 黒線: Content=0 と Content=1 の両方を送信したノード（正規ノードIDが偽コピー記録に混入したケース）
→ 攻撃者の偽装情報が広がる経路を目視で確認可能

### 可視化2: タイムライン図
- X 軸: 時刻 (Time)、Y 軸: ノードインデックス（初登場順）
- 斜め線で各通信の「送信元→送信先」と発生時刻の対応を表現
- 薄い線 (Content=0) と濃い線 (Content=1) で情報種別を区別
- 偽装情報の発生タイミングと拡散の速さを確認できる

## 現状の限界
- 攻撃者の自動判定機能なし（色の違いを目視で確認するのみ）
- 攻撃者特定の定量評価（Precision/Recall/F1）は未実装
→ 次回で対応（20260519.md 参照）

## 2026-05-13 追記: オンライン攻撃者検出ロジックの検討

### 質問の要旨

現在の Method 9 の検出ロジックは「info_id=0 が TRUE、info_id=1 が FAKE」という前提に基づいてノードを分類しているが、**オンライン検出ではどちらが正しい情報か各ノードは知らない**ため、このロジックをそのまま使うことは難しい。

攻撃者は受け取ったすべての経路履歴を改変（info_id を書き換え）することが分かっている。複数の経路履歴を時系列で受信していくなかで、**info_id が一致する部分と一致しない部分の不整合をうまく利用して攻撃者を絞り込めないか？**

### 現状の実装の問題点

`method3.hpp` L835–849 の検出コア:
```cpp
int cnt_0=0, cnt_1=0;
for(const EventMapUnit& em : event_map){
    if(em.send_trust_count.at(0)>0 && em.send_trust_count.at(1)==0){ ++cnt_0; }
    else if(em.send_trust_count.at(0)==0 && em.send_trust_count.at(1)>0){ ++cnt_1; }
}
if(cnt_1>cnt_0){ trust=1; } else if(cnt_1<cnt_0){ trust=0; }
```

- `send_trust_count` = 各ノードが送信した info_id ごとのカウント
- これは「info_id=0 を送っていないノードが多ければ FAKE が多数派」という多数決判断
- オフラインならば多数決で正解が分かるが、**オンラインではどちらが TRUE か分からない**ため本質的に使えない

また、`malicious_disguise()` (L454–487) の実装を見ると、攻撃者はイベント履歴を**まるごとコピーし `information_id` だけ書き換える**ことが確認できる:
```cpp
EventDataReceive ed = share->store.GS->events.at(i);
ed.information_id = trust; // 攻撃者の利用している情報に書き換える
store.GS->append(ed);      // グローバルストアに混入
```

### 提案アルゴリズム

#### Algorithm A: 入出力 info_id 不整合検出（主要案）

**考え方**: 正直なノードは「受信した info_id」と同じ info_id を転送する。攻撃者だけが「受け取ったもの (0) と異なるものを転送する (1)」。この**切り替え行動はグラウンドトゥルースなしで検出可能**。

現在 `EventMapUnit` には `send_trust_count`（送信カウント）しかない。**`receive_trust_count`（受信カウント）を追加**することで以下の判定ができる:

| ノード種別 | receive[0] | receive[1] | send[0] | send[1] |
|---|---|---|---|---|
| 攻撃者 | >0 | >0 (幽霊) | 0 | >0 |
| 正直(TRUEチェーン) | >0 | 0 | >0 | 0 |
| 正直(FAKE受け取り後) | 0 | >0 | 0 | >0 |

攻撃者のみが `receive[0]>0 && send[0]==0 && send[1]>0` という組合せを持つ。

実装上の変更点:
1. `floatingInformation.hpp` の `EventMapUnit` に `receive_trust_count` フィールドを追加
2. `method3.hpp` の `AddProcessReceive2()` で `to_itr` のカウントをインクリメント
3. method 9 の判定ロジックを入出力不整合ベースに置き換え

#### Algorithm B: 幽霊イベント検出（補助案）

攻撃者はコピー+書き換えでイベントを生成するため、グローバルストアに**同一 (from_id, to_id) ペアで info_id が異なる2つのイベント（幽霊イベント）**が混入する。

- 同一エッジに `info_id=0` と `info_id=1` の両方が存在する場合 → 改ざんを検出
- その `from_id` ノード群は攻撃者以前の「正直だが改ざんされた」ノード
- それらの downstream で `send[0]==0 && send[1]>0` のノード = 攻撃者

A と B を組み合わせることで確信度を高められる。

### その他のアイデア

**C. タイムスタンプギャップ分析**:
攻撃者はコピー時に元のタイムスタンプをそのまま引き継ぐ。自身のイベントだけ現在時刻になるため、`receive_times` に「突然新しい時刻に飛ぶ」不連続な箇所が生まれる。

**D. 空間位置整合性チェック**:
`malicious_disguise` は `information_id` だけ書き換え、位置情報 (`send_at`, `at`) はコピーのまま。したがって「同一 (from_id, to_id, timestamp) で位置も同一だが info_id だけ異なる」イベントは改ざんの確実な証拠。位置情報の偽造は困難なため、より強いシグナルになりうる。

**E. 疑惑スコア（連続値信頼度）**:
binary の trust/distrust ではなく各ノードに連続値の疑惑スコアを持たせ、A・B の証拠が積み重なるにつれてスコアを更新する。単一観測のノイズに強い。
