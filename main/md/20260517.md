# 2026-05-17 — ブロックチェーン適用検討

## 概要

進捗報告会にて教授より提案：「ブロックチェーンを使えば経路履歴を攻撃者が改ざんできないのでは？」

現在の攻撃者特定ロジックは、各ノードの送信履歴（`nodes_sent_0`、`nodes_sent_1`）に基づいている。
このログが攻撃者によって改ざんされた場合、検出精度が低下しうる。
ブロックチェーンはそのログ自体の信頼性を保証する技術として候補に挙がった。

---

## 1. ブロックチェーンの改ざん耐性の仕組み

### チェーン構造

新しいブロックを作成するとき、「前のブロックの内容から計算した印（ハッシュ値）」を含める。

```
┌────────────────┐     ┌────────────────┐     ┌────────────────┐
│   ブロック 1    │     │   ブロック 2    │     │   ブロック 3    │
│                │     │                │     │                │
│ Aが時刻100に   │ →印→ │ Bが時刻102に   │ →印→ │ Cが時刻105に   │
│ Content=0 送信 │     │ Content=0 送信 │     │ Content=1 送信 │
│                │     │ 前の印: [h1]   │     │ 前の印: [h2]   │
└────────────────┘     └────────────────┘     └────────────────┘
```

印（ハッシュ値）の性質：**内容が1文字でも変わると、印の値が全く変わる**。

#### 過去を書き換えようとすると…

```
ブロック1を改ざん
  └→ ブロック1の印が変わる
       └→ ブロック2に書いてある「前の印」と不一致 → バレる
            └→ ブロック2も書き換えが必要
                 └→ ブロック3も書き換えが必要
                      └→ … 以降すべてのブロックを書き換え続けなければならない
```

過去の1点を書き換えると、それ以降のすべてが連鎖的に矛盾する。

---

### 分散管理

ブロックチェーンでは、ネットワークの全参加者が同一の記録帳コピーを保持する。

```
        ┌──────────┐
        │ ノード A  │ ─── 記録帳コピー
        └──────────┘
        ┌──────────┐
        │ ノード B  │ ─── 記録帳コピー   ← 全員が同じ内容
        └──────────┘
        ┌──────────┐
        │ ノード C  │ ─── 記録帳コピー
        └──────────┘
              ⋮
```

1人が自分のコピーを書き換えても、他全員のコピーと照合すれば「お前のだけ違う」と即座に検出される。
新しい記録を追加する際は、過半数の参加者が「正しい」と承認したものだけが記録帳に書き込まれる。

## 2. IFへの適用が難しい理由

### 断続接続（DTNの特性）

ブロックチェーンの合意（全員で承認）には、ノード間の通信が前提となる。
しかし情報フローティングの車両ネットワークでは：

```
時刻 t1:    [車A] ←Cr→ [車B]  （すれ違い中：通信可）

時刻 t2:    [車A] →→→→→→→→    [車B]  （走り去った：通信不可）
                     ↑
              もう圏外
```

- 通信可能な時間が非常に短い
- ノードが頻繁に現れては消える
- 全員が同時にオンラインであることを保証できない

そのため、ブロックチェーンの「全員で確認し合う」前提が崩れる。

### 問題の整理

| 項目 | 一般的なブロックチェーン | IFへの適用 |
|------|--------------------------|-----------|
| ネットワーク | 常時接続インターネット | 断続接続・アドホック |
| 合意の取り方 | 全員がオンラインで投票 | 接続が切れていて投票できない |
| 記録の保管 | 全参加者が持つ | 移動する車がどう持ち合うか未解決 |
| 参加者数 | 多いほど安全 | 少数かつ流動的 |

## 3. ブロックチェーンが機能する具体的な場面

### シナリオ：S → A → B → X（Xが攻撃者）

```
S ──(Content=0)──→ A ──(Content=0)──→ B ──(Content=0)──→ X（攻撃者）
                                                           │
                                                    履歴を改ざんして配信
                                                    「S・A・BはContent=1を送った」
                                                           │
                                                           ↓
                                                       下流ノード
```

#### ブロックチェーンなしの場合

下流ノードはXの偽の履歴をそのまま受け取る：

```
受け取ったパケット: S→A→B→X 全員 Content=1 を送信
  → S・A・B・X 全員が攻撃者候補に見える
  → Xだけを特定できない
```

#### ブロックチェーンありの場合

全員が保持する記録帳と照合できる：

```
【ブロックチェーンの記録（改ざん不可）】
  S の記録: Content=0 を送信  ✓
  A の記録: Content=0 を送信  ✓
  B の記録: Content=0 を送信  ✓
  X の記録: Content=1 を送信  ✓

Xが主張する履歴: S・A・B が Content=1 を送信
                       ↕ 矛盾
ブロックチェーンの記録: S・A・B は Content=0 のみ
  → X の主張は嘘 → X が攻撃者と特定
```

ブロックチェーンにより、`nodes_sent_0` と `nodes_sent_1` の集合そのものが改ざん不可能な証拠となる。

---

## 4. まとめ（研究上の位置づけ）

ブロックチェーンは「経路履歴ログの信頼性保証」レイヤーであり、攻撃者特定ロジック自体（`nodes_sent_1 - nodes_sent_0`）は変わらない。
**提案手法の実用化条件**として位置づけられる。

ただし、車両ネットワーク（DTN）への適用には断続接続という根本的な課題があり、現在も研究が進行中のオープンな問題である。
本研究では「将来課題」として言及する範囲で十分。
